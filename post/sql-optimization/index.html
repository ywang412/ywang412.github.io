SQL 1 - Optimization and Best Practice

2018-10-03

Best practice

  1. use "=" not "<>" on index
  2. use Limit 1 if only return 1 record
  3. use TINYINT before SMALLINT and INT
  4. break big DELETE, UPDATE or INSERT into small ones
  5. use UNION ALL instead of UNION because ALL allows duplicate
  6. don't use select * which will not use index
  7. add index to where, join, order by
  8. use limit to return page

Order by and Group by

mySql order by use two kinds of sorting. 1) using index 2) using filesort

If data size < sort_buffer_size, sort in memory (number of tmp files 0), else in disk (number of tmp files > 0).

Filesort has 3 modes. <sort_key, rowid> only sort rowid and fetch records after max_length_for_sort_data < length of all fields <sort_key, additional_fields> loads all fields and sort max_length_for_sort_data > length of all fields <sort_key, packed_additional_fields>

  1. Add index to sort by field
  2. order by (a , b) use index (a, b) not (b, a)
  3. where a = 1000 order by b use index (a, b)
  4. select * from t1 order by a,b; won't use index because scan index and fetch all records costs more than filesort. select id,a,b from t1 order by a,b;
  5. explain select id,a,b from t1 where a>9000 order by b; won't use index because in range a > 9000, data is not sorted by b. it is sorted by (a, b)
  6. explain select id,a,b from t1 order by a asc,b desc; won't use index

Group by can use "order by null" to disable sorting.

SQL Paging

select * from t1 limit 99000,2;

  1. use auto-increment continuous primary key select * from t1 where id >99000 limit 2;

select * from t1 order by a limit 99000,2; won't use index because scan index and fetch all records costs more than filesort. 2. return id when sort select * from t1 f inner join (select id from t1 order by a limit 99000,2)g on f.id = g.id;

Rewrite select * from t1 limit 99000,2; to select * from t1 as a join (select id from t1 limit 99000, 2) as b on a.id = b.id;

SQL join

Index Nested-Loop Join if join on index key. Small table will be the driving table. T ~ 2*N(small table).

Block Nested-Loop join if join on key without index. Load small table into join_buffer. Scan each row of big table and compare it to each row in join_buffer. Total number of scanned row is M(big table) + N(small table). T ~ M(big table)*N(small table).

  1. join on index key
  2. use small table as driving table select * from t2 straight_join t1 on t2.a = t1.a;
  3. temporary table Add index to field b in temp table
CREATE TEMPORARY TABLE `t1_tmp` (
  `id` int(11) NOT NULL AUTO_INCREMENT,
  `a` int(11) DEFAULT NULL,
  `b` int(11) DEFAULT NULL,
  `create_time` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '记录创建时间',
  `update_time` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '记录更新时间',
  PRIMARY KEY (`id`),
  KEY `idx_a` (`a`),
  KEY `idx_b` (b)
) ENGINE=InnoDB ;

insert into t1_tmp select * from t1;
select * from t1_tmp join t2 on t1_tmp.b= t2.b;

Count (*)

Count(a) will not count null but Count(*) will. Count(1) and Count(*) will be the same. MyISAM has meta information about total number of rows. InnoDB doesn't have the meta information because it allows concurrent transactions to have different row count.

MySql will use smallest non-clustered index to calculate count(*). If there is no non-clustered index, it will use clustered-index. Because non-clustered index (secondary) only store primary key while clustered-index (primary key) stores entire row.

  1. show table status like 't1'; will give an estimate of rows.

  2. use redis
